단일 로그인 · 역할 라우팅 설계

로그인은 하나. 로그인 후 역할(Role)에 따라 자동으로 각자의 화면으로 이동합니다.

내부 설계 문서 · noindex · 2026-07-21 · 기존 B2B 스키마(organization_members / child_guardians / chewstep_staff) 위에 설계

1핵심 원칙

  • 계정은 하나의 accounts 테이블 — Supabase auth.users + profiles. 부모·교사·원장·기관보호자 모두 같은 계정 체계.
  • 로그인도 하나 — chewstep.ai 우측 상단 “로그인” 하나. 사용자는 부모 사이트/기관 사이트를 구분할 필요가 없습니다.
  • 역할이 화면을 결정 — 로그인 후 role을 확인해 자동 이동. 결국 화면만 다릅니다.
  • 역할 권위는 membership 테이블에만 — 한 사람이 여러 역할일 수 있어(교사이면서 학부모) 단일 enum 금지. profiles.user_type은 “기본 진입화면 힌트”로만 사용(ERD 결정 C1).

3기관 등록 보호자 — 초대 딥링크 흐름

보호자는 “기관 사이트인지 부모 사이트인지” 전혀 신경 쓰지 않습니다. 문자의 링크만 누르면 로그인 후 자기 아이 리포트가 바로 열립니다.

① 초대 문자(SMS)“Chewstep 식사 리포트가 도착했습니다. [리포트 보기]”
→
② 링크 클릭chewstep.ai/login?next=/app/report/서연
→
③ 로그인같은 로그인 하나. 처음이면 가입.
→
④ 자동 이동role=기관보호자 확인 → 서연이 리포트가 바로 열림
딥링크 규칙: 로그인 페이지는 ?next= 파라미터를 보관했다가 로그인 성공 후 그 경로로 이동합니다. next가 없으면 role 기본 화면으로 이동. (같은 도메인 경로만 허용 — open-redirect 방지)

5역할 리졸버 — get_my_landing() RPC 권장

브라우저에서 여러 테이블을 직접 조회하면 RLS·성능·노출이 복잡해집니다. 로그인 사용자의 역할·이동경로를 한 번에 돌려주는 SECURITY DEFINER RPC를 두는 것이 SaaS 표준입니다.

-- 로그인한 사용자(auth.uid())의 역할과 기본 이동 경로를 반환
create function get_my_landing() returns jsonb
language plpgsql security definer set search_path=public as $$
declare uid uuid := auth.uid(); r text; route text;
begin
  if uid is null then return jsonb_build_object('role','anon','route','/login'); end if;
  -- 1) 운영자
  if exists(select 1 from chewstep_staff s join profiles p on p.id=s.profile_id
            where p.auth_user_id=uid and s.status='active') then
    return jsonb_build_object('role','staff','route','/admin'); end if;
  -- 2) 원장 / 3) 교사  (organization_members.role)
  select m.role into r from organization_members m join profiles p on p.id=m.profile_id
    where p.auth_user_id=uid and m.status='active'
    order by (m.role='director') desc limit 1;
  if r='director' then return jsonb_build_object('role','director','route','/director'); end if;
  if r='teacher'  then return jsonb_build_object('role','teacher','route','/teacher'); end if;
  -- 4) 기관 등록 보호자
  if current_guardian_id() is not null and exists(
       select 1 from child_guardians where guardian_id=current_guardian_id()) then
    return jsonb_build_object('role','guardian','route','/app/report'); end if;
  -- 5) 일반 부모(기본)
  return jsonb_build_object('role','parent','route','/app');
end $$;

프론트: 로그인 성공 → rpc('get_my_landing') 호출 → next ?? route로 location.replace(). 헬퍼 current_guardian_id()는 이미 존재. 컬럼명(profiles.auth_user_id 등)은 실제 스키마로 확정 후 배포.

6확정이 필요한 것

  • 완료 계정 = B2B 운영으로 일원화 결정. get_my_landing() RPC를 B2B 운영에 배포하고 login.html을 실제 인증에 연결(로그인→역할판정→이동, 딥링크·open-redirect 방지 e2e 통과). 남은 것: B2C 데모 계정 이관 범위, 신규가입(보호자 초대 redeem) 플로우.
  • 결정 미구현 화면 — 교사(우리반)·원장(기관 관리자) 대시보드는 신규 제작 필요. 어디부터 만들지.
  • 결정 딥링크 파라미터 형식 — ?next= 경로 방식 vs 리포트 토큰(?token=) 방식. 문자 링크 보안·만료 정책 포함.
